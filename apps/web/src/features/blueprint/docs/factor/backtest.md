## 策略配置 → 保存 → 部署

回测不仅回答"策略能赚钱吗", 还承担**策略定义**功能:

### 策略实体定义

| 字段 | 说明 |
|---|---|
| 策略名 | 用户自定义名称 (e.g. "小盘动量v2") |
| 因子组合 | 从 Library 选取 N 个因子 |
| 因子权重 | 等权 / **IC加权 (推荐)** / 手动权重 |
| 调仓频率 | 日 / 周 / 月 |
| 选股规则 | Top N / Bottom N / 多空 |
| 风控规则 | 最大持仓数 / 单股上限 / 行业上限 |
| 基准 | 沪深300 / 中证500 / 中证1000 |
| 滑点模型 | Level 0 / Level 1 / Level 2 (见 expand) |

**因子依赖约束**:
- 策略引用的因子必须是 LIVE_ACTIVE 或 PAPER_TEST 状态
- 因子进入 PROBATION → 策略标记警告 "底层因子健康异常"
- 因子 RETIRED → 强制重新选择因子, 策略不能使用已退役因子

### 工作流

1. **配置**: 工具栏配置策略参数
2. **回测**: 运行回测验证策略有效性
3. **保存**: 回测通过 → "保存为策略" 按钮 (工具栏右上角) → 弹出保存对话框 (策略名必填 + 备注选填 + 基准确认) → 策略持久化 (状态=Draft)
4. **部署**: 策略通过审核 → "部署到 Book" → 选择目标 Book → 分配资金 → 上线 (状态=Live)

### 策略生命周期

`Draft → Testing → Validated → Paper → Live → Degrading → Retired`

保存的策略在 Portfolio/Books 中可见, 作为 Book 的"Strategy"组件。策略实体的完整定义参见 Home tab 概念索引 — Factor → Strategy → Book 关系。

## 执行假设 — A 股交易模型

**交易时点**: 默认 T+1 日开盘集合竞价 (9:15-9:25 非连续撮合)

### 滑点模型 (3 级)

| Level | 名称 | 模型 | 用途 |
|---|---|---|---|
| **Level 0** | 零滑点 | 无交易成本 | 仅验证因子预测力, **不代表实际收益** |
| **Level 1** | 固定滑点 | 单边 0.1% (含佣金+印花税+冲击估算) | **默认, MVP** |
| **Level 2** | 成交量比例滑点 | 冲击 = k × √(交易量/日均成交量), k=0.1 | P2, 需个股日均成交量数据 |

### A 股交易成本明细

| 成本项 | 费率 | 说明 |
|---|---|---|
| 佣金 | 万 2.5 (双边) | 买卖各收一次 |
| 印花税 | 千 1 (卖出) | 仅卖出时收取 |
| 过户费 | 万 0.1 | 沪市专有, 深市无 |
| 单边合计 | 约万 3.6 | 佣金+印花税+过户费 |

### 涨跌停处理

- 涨停日无法买入 → 延后到下一个可交易日
- 跌停日无法卖出 → 标记为挂单中, 次日优先执行
- 回测引擎需维护 "挂单队列"

### T+1 约束

- 买入当日不可卖出, 回测引擎必须实现持仓日期追踪
- 影响: 极端行情下无法止损, 策略需考虑隔夜风险

**Tier 标注**: Level 0-1 = MVP, Level 2 = P2, 市场微观结构模型 = P3

## A 股公司行为处理

与 Lab tab 的处理规则一致 (参见 Lab — A 股公司行为处理), 回测特有规则:

| 公司行为 | 回测处理 |
|---|---|
| **退市** | 退市日以最后交易价结算, **计入组合损失** (不删除, 否则引入生存偏差) |
| **停牌复牌** | 停牌期间视为持仓冻结, 复牌后按复牌价重新计算净值 |
| **生存偏差声明** | 回测报告自动标注 "样本宇宙: 含/不含退市股" 及对 Sharpe 的影响估算 |

## 回测引擎架构

- **后端**: Python 向量化回测引擎 (非事件驱动), 基于 numpy/pandas
- **回测流程**: 因子计算 → 排序 → 选股(Top/Bottom N) → 等权/加权 → 逐日计算净值 → 统计指标
- **样本分割**: 默认 70/30 (样本内/样本外), 支持自定义比例
- **基准数据**: AKShare `stock_zh_index_daily()` 获取沪深300等指数日线

### 防未来函数

- 严格使用 T 日因子值选 T+1 持仓, 确保无前视偏差 (look-ahead bias)
- 回测报告自动检测并标记潜在的前视偏差风险

### 水下曲线计算

- Drawdown = (当前净值 - 历史最高净值) / 历史最高净值
- 最大回撤 = max(Drawdown序列)
- 恢复天数 = 从最大回撤点到净值重新创新高的交易日数
- 未恢复标注: 如果到回测结束仍未恢复, 显示"未恢复"标签

### K线标注实现 (P2)

- D3 OHLC rendering + signal overlay markers
- 买卖点标注层: 独立 SVG layer 叠加在 K 线图上
- 借鉴 VectorBT 的可视化风格: 买入绿色三角, 卖出红色三角, 持仓区间半透明背景色

## 回测配置详解

| 参数 | 选项 | 说明 |
|---|---|---|
| 因子选择 | 从Library选取 | 支持多因子组合 |
| 组合方式 | 等权 · **IC加权 (推荐)** · 手动权重 | IC 加权考虑因子预测力方向 |
| 调仓频率 | 日 · 周 · 月 | 影响换手成本 |
| 回测区间 | 起始日~结束日 | 含样本内/外分割 |
| 基准 | 沪深300 · 中证500 · 中证1000 | 3 种基准 |
| 滑点模型 | Level 0 · Level 1 · Level 2 | 默认 Level 1 |

> **P2 研究方向 — ML 经验加权**: 使用 ML 模型的特征重要性 (feature importance) 作为因子权重, 公式 `score = Σ(k_i × t_i × x_i)` 其中 k=IC 方向, t=ML 特征重要性, x=标准化因子值。论文实证: AdaBoost/随机森林经验加权显著优于等权 (Sharpe 从 0.05 提升到 0.88-1.01)。需 ML 模型月度重训练。

## 策略配置 → 保存 → 部署流程

### 策略实体 (Strategy)

策略 ≠ 因子。策略 = N 个因子 × 权重 + 调仓频率 + 选股规则 + 风控规则。完整实体定义参见 Home tab 概念索引。

### 保存流程

1. 用户在配置栏完成策略参数配置
2. 点击"运行回测" → 查看净值/指标/归因
3. 回测结果满意 → 点击"保存为策略"按钮 (工具栏右侧)
4. 弹出保存对话框: 策略名 (必填) + 备注 (选填) + 基准确认
5. 保存后策略状态为 Draft, 进入 Portfolio/Books 可见列表

### 部署到 Book 流程

1. 在 Portfolio/Books 中创建或选择 Book
2. Book 配置页 → "添加策略" → 从已保存策略列表中选择
3. 分配资金: Book 总资金 × 该策略权重
4. 设置风控: 继承策略风控规则 + Book 级别风控叠加
5. 审批: 需要管理员确认 (防止未验证策略上线)
6. 部署: 策略状态 Draft → Live, Book 开始产生交易信号

### 策略与 Book 的关系

- 1 个 Book 可包含多个 Strategy (多策略混合)
- 1 个 Strategy 可被多个 Book 引用 (共享信号, 独立资金)
- Book = Strategy + Capital + Broker + Risk Rules (参见 product-vision.md)

> **工程备注 — 回测引擎分期**:
> - **MVP (4-6 周)**: 向量化净值计算 + Level 0/1 滑点 + QuantStats 指标 + 基础净值曲线
> - **P2 (4-6 周)**: Brinson 归因 + Walk-Forward + Level 2 滑点 + 月度热力图 + K 线标注
> - **P3 (4-6 周)**: 参数优化 + 容量分析 + 实时模拟盘
> - 不建议一期全做。回测引擎的质量 (防未来函数/滑点模型) 远比功能数量重要。

## KPI 仪表盘详解

### QuantStats 集成

- 使用 quantstats Python 库一键生成策略体检报告
- 输出: HTML 报告 + JSON 指标, 前端展示 JSON 指标
- 指标覆盖: 年化收益/波动率/Sharpe/Sortino/Calmar/MaxDD/胜率等 20+ 项

### 指标判定阈值

| 指标 | 优秀 | 达标 | 不达标 |
|---|---|---|---|
| Sharpe | > 2.0 | > 1.0 | < 1.0 |
| Sortino | > 2.5 | > 1.5 | < 1.5 |
| Calmar | > 1.5 | > 0.8 | < 0.8 |
| MaxDD | < 10% | < 20% | > 20% |
| 胜率 | > 60% | > 50% | < 50% |

> **阈值来源**: 参见 quant-domain.md 绩效评估阈值。

### P2: 过拟合校正指标 (PSR / DSR)

> 来源: López de Prado, Lipton, Zoonekynd (2026) "How to Use the Sharpe Ratio" — quant-domain/factor-analysis/evaluation.md

当用户运行 **10+ 次回测**后, 以下指标自动激活:

| 指标 | 含义 | 阈值 | 计算依据 |
|---|---|---|---|
| **PSR** (概率夏普比率) | Sharpe 真正 > 基准 SR₀ 的概率 | > 95% = 可信 | 基于 Sharpe 采样分布, 考虑偏度/峰度/自相关 |
| **DSR** (减偏夏普比率) | 修正多重测试后的 Sharpe | > 1.0 = 有效 | 根据测试次数 K 和所有 SR 横截面方差 "脱水" |

**PSR 惩罚因素**:
- 负偏度 (下行风险) → PSR 下降
- 正超额峰度 (肥尾) → PSR 下降
- 正序列相关 (波动率聚集) → PSR 下降

**DSR 激活逻辑**:
- 系统追踪用户对同一策略的所有回测尝试次数 K
- K < 10: 仅显示 Sharpe, PSR/DSR 灰色不可用 + tooltip "至少需要 10 次回测才能激活过拟合校正"
- K ≥ 10: PSR/DSR 自动计算并显示在 KPI 面板末尾
- DSR 对最高 Sharpe "脱水": 尝试越多, 脱水越多

**Tier 标注**: P2 — MVP 仅展示 Sharpe/Sortino/Calmar 等传统指标。PSR/DSR 需要 Bootstrap 采样分布估计 + 多次回测记录追踪。

### 可视化设计

- 每个 KPI 独立卡片: 指标名 + 数值 + 状态图标 + 基准对比
- 颜色编码: 绿色(优秀) / 黄色(达标) / 红色(不达标)

## 深度分析详解

### Brinson 归因公式

- **配置效应**: `Sum((wp - wb) × Rb)` — 板块权重偏离 × 基准板块收益
- **选股效应**: `Sum(wb × (Rp - Rb))` — 基准权重 × 板块超额收益
- **交互效应**: `Sum((wp - wb) × (Rp - Rb))` — 权重偏离 × 超额收益

归因结果表格列定义:

| 列 | 说明 |
|---|---|
| 行业名 | 申万一级行业 |
| 配置效应 | 权重偏离贡献 |
| 选股效应 | 选股超额贡献 |
| 交互效应 | 权重×超额交叉项 |
| 合计 | 三项之和 |

### 月度热力图实现

- D3 heatmap 或 ngx-charts heat-map, 12xN grid
- 色阶: 亏损红色梯度 → 白色(零) → 盈利绿色梯度
- hover 显示: 月度收益率 + 基准收益率 + 超额
- 边框汇总: 最右列 = 年化收益, 最底行 = 月均收益

### Walk-Forward 详解

- **目的**: 防止过拟合的最佳实践, 模拟真实投资中只能用历史数据的约束
- **流程**: 12 个月训练 → 3 个月测试 → 窗口向前滑动 3 个月 → 重复
- **评估**: 各窗口 OOS Sharpe 的均值和标准差
- **比率解读**: IS/OOS Sharpe > 2.0 → 严重过拟合嫌疑, < 1.5 → 可接受
- **可视化**: 滚动窗口 Sharpe 折线图, 每个窗口 IS vs OOS 柱状图对比
- **可配置**: 训练窗口 (6/12/24 月) + 测试窗口 (1/3/6 月) + 滑动步长 (1/3 月)

### 参数热力图实现

- 2D grid 搜索, 对每个参数组合运行回测
- 参数选择建议: 避免过多参数(<=3个), 每个参数<=10个取值
- 可视化: ngx-charts HeatMap, X/Y 轴为参数值, 色阶为 Sharpe/收益率
- **关键判断**: "平原"=参数微调对结果不敏感(稳健), "尖峰"=结果高度依赖特定参数(过拟合)
- **参数轴选择**: 下拉菜单让用户选择哪两个参数展示 (例如调仓频率 × 持股数量)

### 容量估算方法 (P2)

- 基于日均成交量, 假设可捕获 1-5% 流动性
- 不同 AUM 下模拟: 冲击成本随规模非线性增长
- 输出: Sharpe vs AUM 衰减曲线
- **局限声明**: 容量估算高度依赖市场冲击模型的准确性, MVP 阶段仅提供粗略估算
